1. ПРИНЯТЫЕ АРХИТЕКТУРНЫЕ РЕШЕНИЯ

1.1 Хранение данных: JSON → SQL (MySQL)

Было: JSON-файлы на Яндекс.Диске, отдельный файл на каждого участника
Стало: MySQL (есть бесплатно на Timeweb)
Основания:
Составные ключи решают race condition штатно (транзакции)
YD API: latency 200–500 мс, лимит 10 req/sec — не годится для БД
SQL масштабируется, JSON — нет
Одна таблица
votes
с PK
(p_num, q_num, u_code)
вместо 30 файлов

1.2 Яндекс.Диск — только файловое хранилище

PDF протоколов, материалы, особые мнения, бэкапы БД
Не для структурированных данных

1.3 Среда разработки: VDS + Python (Flask)

VDS Timeweb (~179–300 ₽/мес) — одна оплата за всё
На одном VDS: nginx + Flask + MySQL + webhook + cron
PHP-порт для тиражирования — потом, когда продукт стабилен
Основания: Python знаком, Flask уже частично написан, переписывание на PHP = 3 недели ради экономии 50 ₽/мес

1.4 PWA — вместо мобильного/десктопного приложения

Два файла:
manifest.json
+
sw.js
(~1 час работы)
Покрывает телефон + десктоп + тиражируемость
Добавляется в конце, не увеличивает объём

1.5 Локальная разработка → GitHub → VDS

Вся отладка — на локальном компьютере (Mac)
git push
→ GitHub (версионирование)
Перенос на VDS — один раз, когда логика откатана
Автодеплой (webhook) — только на VDS, локально не нужен
Файл
config/env.py
— разделение сред
local
/
production
